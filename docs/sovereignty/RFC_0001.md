# RFC 0001: Constitutional Interface Specification for Human-AI Interaction

## Abstract

This document defines a portable constitutional interface for human-AI interaction. It introduces two orthogonal control axes for agentic workflows: `Form` (constraining the permissible speech act or execution type) and `Velocity` (constraining the execution pacing and continuation state). By exposing these constraints to the model as first-class, machine-readable metadata—and enforcing them via a strict ratification layer—systems can achieve verifiable, governable interaction across chat, voice, and API surfaces. The goal of this specification is not better prompting; the goal is cryptographically auditable, structurally governed AI interaction.

---

## 1. Introduction and Motivation

Current AI interfaces rely predominantly on free-form prompting combined with procedural UI controls (e.g., dropdowns, required fields, and branching logic). While these mechanisms successfully constrain the human user, they fail to constrain the underlying model as a first-class participant in the interaction. 

This structural gap creates four recurring failure modes in enterprise and sovereign AI deployments:
1. The governing interaction shape is not consistently communicated to the model's attention mechanism.
2. The same constraints must be rebuilt from scratch per modality (web, voice, API).
3. Constitutional violations manifest as standard UX errors rather than auditable security events.
4. The boundary between *recommending* an action and *executing* an action is easily blurred by the model.

This specification addresses these vulnerabilities by defining a minimal, portable grammar for interaction shape, enforced at the execution boundary.

## 2. Conventions Used in this Document

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

## 3. Core Principles and Interaction Model

The fundamental principle of this specification is that **Form and Velocity are orthogonal**. A system MAY independently reject, relabel, pause, escalate, or stop an interaction on either axis.

A compliant constitutional interaction is defined as the following tuple:

```json
Interaction := {
  "utterance": "string",
  "form": "string",
  "velocity": "string",
  "authority": "string",
  "context": "object"
}
```

Where:
*   `utterance`: The natural-language payload.
*   `form`: The active interaction type (epistemic/action state).
*   `velocity`: The active control-state signal.
*   `authority`: The actor or layer permitted to advance, defer, or halt the interaction.
*   `context`: Persistent state including defaults, history, inferred values, and cryptographic receipts.

## 4. The Form Axis

The `Form` axis dictates the permissible epistemic posture and operational bounds of the output. 

### 4.1. Conforming Forms
A compliant system MUST support the following minimal forms:
`FACT`, `HYPOTHESIS`, `ASSUMPTION`, `QUESTION`, `RECOMMEND`, `EXECUTE`.

### 4.2. Form Semantics
*   **`FACT`**: Used for verifiable claims. Speculation MUST NOT be presented as established truth.
*   **`HYPOTHESIS`**: Used for plausible but unverified claims or exploratory reasoning. Uncertainty MUST be explicit.
*   **`ASSUMPTION`**: Used for temporary premises within a bounded reasoning scope. MUST NOT be silently exported as fact.
*   **`QUESTION`**: Used for clarification or missing inputs. SHOULD collapse to a non-`QUESTION` form once inputs are supplied. SHOULD NOT persist across turns without reassertion.
*   **`RECOMMEND`**: Used for advice, options, or plans. MUST remain strictly distinct from execution.
*   **`EXECUTE`**: Used for operations that commit, mutate, transact, or dispatch. MUST satisfy external approval, policy, and authority constraints prior to continuation.

## 5. The Velocity Axis

The `Velocity` axis dictates the pacing, suspension, or termination of the execution pipeline.

### 5.1. Conforming Signals
A compliant system MUST support the following velocity signals:
`PROCEED`, `PAUSE`, `ESCALATE`, `STOP`.

### 5.2. Precedence and State Hierarchy
Velocity resolution MUST follow strict precedence:
`STOP > ESCALATE > PAUSE > PROCEED`

### 5.3. Velocity Semantics
*   **`PROCEED`**: Continue within current form, policy, and authority bounds.
*   **`PAUSE`**: Suspend generation/execution and await further input.
*   **`ESCALATE`**: Defer to a higher-authority actor or review layer. Substantive execution MUST halt.
*   **`STOP`**: Terminate the active interaction or action scope completely. 

*Implementation Note: If effective velocity cannot be established due to ratification delay, transport failure, or governance-layer fault, the system MUST default to `PAUSE`.*

## 6. Authority and The Ratification Rule

Authority is a distinct governing dimension adjacent to velocity. 

### 6.1. Authority Precedence
`CUSTODIAN > POLICY > ADVISORY`
*(Custodian = Sovereign human/keyholder; Policy = System constraint layer; Advisory = Model recommendation)*

### 6.2. The Ratification Rule (Security Critical)
Model-emitted velocity signals are strictly **advisory** until ratified by the constitutional policy or custodian layer. 

A compliant system MUST distinguish between `model_recommended_velocity` and `effective_velocity`.
*   The model MAY recommend `PAUSE` or `ESCALATE`.
*   The model MUST NOT unilaterally clear `STOP`, clear `ESCALATE`, or promote itself to the controller of execution flow.

## 7. Lifecycle and Inference

### 7.1. State Machine Lifecycle
A compliant interaction MUST pass through the following lifecycle:
`RECEIVE -> NORMALIZE -> VALIDATE -> EXECUTE -> REVIEW -> EMIT`

### 7.2. Inference and Transparency
Systems MAY infer `Form` from phrasing or modality (e.g., inferring `EXECUTE` from "Transfer funds"). If inferred:
*   The inferred form MUST be preserved in logs and receipts.
*   The explicit user form MUST override the inferred form unless blocked by policy.
*   Product implementations SHOULD expose inferred forms to the user in real time.

### 7.3. Lifecycle Transition Rules
*   Missing required inputs SHOULD transition to `QUESTION` + `PAUSE`.
*   Policy conflict SHOULD transition to `ESCALATE` or `STOP`.
*   Epistemic mismatch SHOULD trigger `RELABEL`, `BLOCK`, `ESCALATE`, or `STOP`.
*   Unauthorized `EXECUTE` commands MUST NOT proceed.

## 8. Enforcement and Transport Integrity

### 8.1. Boundary Enforcement
*   **Input Enforcement**: The system MUST reject malformed metadata, attempts to bypass `STOP` states, or execution attempts lacking sufficient authority.
*   **Output Enforcement**: The system MUST block speculative output under `FACT`, block execution under `RECOMMEND`, and interrupt streams crossing a velocity boundary.
*   **Zero Tolerance Rule**: Silent tolerance of detected violations SHOULD NOT occur. Violations MUST trigger visible friction (e.g., `BLOCK`, `RELABEL`).

### 8.2. Transport Integrity
Constitutional metadata MUST survive intermediary transforms (speech-to-text, orchestration layers, API gateways). Transport layers MUST NOT strip, silently rewrite, or drop active form, effective velocity, or authority metadata without generating an auditable security event.

## 9. Persistence and Audit

Receipts SHOULD make constitutional events reconstructable. A compliant system SHOULD persist:
Normalized form, effective velocity, authority level, signal emitter, model-recommended velocity, inferred vs. explicit form, blocked/relabeled outputs, timestamps, and the final emitted output.

## 10. Reference Flows

**10.1. Fact Query**
```text
Input:  [FORM=FACT] [VELOCITY=PROCEED] "What is the capital of France?"
Output: [FORM=FACT] "Paris is the capital of France."
```

**10.2. Missing Inputs**
```text
Input:  [FORM=EXECUTE] [VELOCITY=PROCEED] "Book my flight."
Output: [FORM=QUESTION] [VELOCITY=PAUSE] "What origin, destination, and date should I use?"
```

**10.3. Recommendation vs. Execution**
```text
Input:  [FORM=RECOMMEND] [VELOCITY=PROCEED] "How should I rebalance my budget?"
Output (Allowed): "A ranked plan with tradeoffs."
Output (Disallowed): "I have moved the funds."
```

**10.4. Ratified Escalation**
```text
Input:  [FORM=FACT] [VELOCITY=PROCEED] "Did production violate policy yesterday?"
Model Advisory: ESCALATE
Output (Effective): [VELOCITY=ESCALATE] "Escalated for review due to insufficient verified evidence."
```

## 11. Non-Goals and Open Questions

**Deferred to v1.0 / Out of Scope:**
*   Training methods for model compliance.
*   Cryptographic implementation details for Sovereign Authority signatures.
*   Multi-agent arbitration semantics.
*   Whether `EXECUTE` should support standard subtypes (e.g., `TRANSACT`, `MUTATE`) or remain flat.
*   Whether a distinct `risk_tier` should be added as a third orthogonal control field.
